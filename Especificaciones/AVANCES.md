### 10.1 — Presets de proveedor en el portal (Config LLM)
- **Objetivo:** reducir errores al crear una ConfigLLM estandarizando endpoint/auth por proveedor.
- **Alcance (solo frontend):** en `src/ElTejido.Web/.../features/config-llm/config-llm.page.ts`, reemplazar el input libre de `proveedor` por un `<select>` (AzureOpenAI / OpenAI / OpenRouter / Anthropic-via-OpenRouter / Otro) que prerellene `endpoint` y `proveedor` (editables). Hint del formato de `modelo` por proveedor. No cambia backend ni contrato.
- **Aceptacion:** elegir un preset prerellena endpoint; se puede sobrescribir; crear ConfigLLM OpenRouter/OpenAI/Azure sigue funcionando. tsc+prettier verdes; e2e del portal si aplica.

### 10.2 — Que el estado `inactivo` de una ConfigLLM se respete al evaluar
- **Objetivo:** que desactivar una ConfigLLM sea un interruptor real, no solo cosmetico (hoy `ObtenerConfigLlmAsync` resuelve por id **sin filtrar estado**, asi que una campania que la referencia la sigue usando aunque este `inactivo`).
- **Alcance (backend):** en `OrquestadorConversacion` (resolucion de snapshots, ~linea 160-168), tratar `configLlm.Estado != Activo` como no disponible -> fallback seguro (`08 §6`, retro neutra + `evaluacionPendiente`) + `LogSeguridad`. Mismo criterio evaluar para **prompt no aprobado/inactivo** y **rubrica inactiva** (hoy se usa la ultima version sin verificar estado; ver `SUPUESTOS.md#orquestador-conversacional`). Decidir y **registrar el supuesto** (bloquear vs solo avisar).
- **Aceptacion:** unit test: campania con ConfigLLM `inactivo` -> NO llama al LLM, cae a fallback neutro. Build/test/format verde.

### 10.3 — Editar el LLM (y refs) de una campania desde el portal
- **Objetivo:** cambiar el `configLlmRef` (y opcional rubricaRef/promptRefs) de una campania sin recrearla. El backend ya lo soporta (`PUT /api/admin/campanias/{id}`, `ActualizarCampaniaAsync`, acepta `ConfigLlmRef`); falta exponerlo en el portal.
- **Alcance (frontend):** en `features/campanias/campanias.page.ts`, agregar modo edicion que use `PUT` con los mismos `<select>` de creacion (config LLM activa, rubrica, prompts). Agregar `actualizarCampania` en `core/admin-api.service.ts` si falta (con CSRF). Las evaluaciones futuras usan la nueva config; las pasadas conservan su snapshot.
- **Aceptacion:** editar una campania y cambiar su ConfigLLM persiste y se refleja; e2e del portal actualizado. tsc+prettier verdes.

### 10.4 — Adaptador nativo de Anthropic en `LlmClientHttp`
- **Objetivo:** usar una API key de Anthropic directamente (sin pasar por OpenRouter).
- **Alcance (infrastructure):** en `src/ElTejido.Infrastructure/Llm/LlmClientHttp.cs`, agregar una rama por `proveedor` (p. ej. `Anthropic`): endpoint `/v1/messages`, headers `x-api-key` + `anthropic-version`, cuerpo Anthropic (`system` separado de `messages`, `max_tokens`), y parsear `content[0].text`. La salida debe seguir cumpliendo el contrato JSON del evaluador (`08 §4`): como Anthropic no usa `response_format`, el prompt pide JSON explicito y el evaluador valida (ya lo hace, con fallback seguro). La key se sigue resolviendo por `ISecretProvider` (`apiKeyRef`), nunca en logs.
- **Contrato/doc:** sin cambio de contrato; documentar el valor de `proveedor` esperado para Anthropic nativo en `02 §6`/`08`.
- **Aceptacion:** unit tests del cliente con `HttpMessageHandler` fake para la forma Anthropic (auth header correcto, ruta `/v1/messages`, parseo de `content[0].text`); el evaluador valida JSON. Build/test/format verde.

## Despliegue y operacion en Azure (tropiezos reales resueltos 2026-06-15)
> Aprendizajes del primer despliegue real. Si vuelven a aparecer, este es el remedio. Tambien se reflejan en `Guias_Implementacion/Guia_Azure_Portal_Paso_a_Paso.md`.
- **CD (`deploy.yml`, OIDC):** push a `main` despliega. El subject del token OIDC debe coincidir con la credencial federada. La guia crea la credencial como **Branch=main** (`repo:ORG/REPO:ref:refs/heads/main`), por eso el job **NO** usa `environment:` (con environment el subject seria `...:environment:production` y da `AADSTS700213`). Para reactivar gate de aprobacion: agregar `environment` + credencial federada de tipo Environment.
- **`No subscriptions found`** en `azure/login` = la App Registration no tiene rol RBAC. Asignar **Contributor**/Website Contributor al SP `gh-eltejido-deploy` sobre el resource group (guia §9.3). Solo con app+credencial no basta.
- **Hostname unico del App Service:** los App Service nuevos NO usan `<name>.azurewebsites.net`; usan `<name>-<hash>.<region>-01.azurewebsites.net` (ej. `app-eltejido-mvp-evd8ffcgd3fthshw.eastus-01.azurewebsites.net`). El smoke test usa la URL real (`steps.deploy.outputs.webapp-url`).
- **Cosmos rol de DATOS:** leer/escribir documentos con managed identity usa el rol de plano de datos **Cosmos DB Built-in Data Contributor** (a veces invisible en IAM del portal; asignar por `az cosmosdb sql role assignment create ... --role-definition-id ...0002`, guia §7.4). El rol de control "DocumentDB Account Contributor" NO sirve para datos.
- **Unique key de `users`:** debe ser **`/whatsappNormalizado`** (o ninguna; la app valida unicidad en codigo). Si se pone por error `/pk` (la misma que la particion), TODO segundo usuario falla con `Conflict (409)` porque `pk='usuario'` se repite. Las unique keys de Cosmos son inmutables: hay que **borrar y recrear** el contenedor. El backend ahora traduce ese 409 a `409 CONFLICT` limpio (no a 500), pero el remedio es corregir el contenedor.
- **TTL en `security` y `leases`:** Data Explorer -> contenedor -> Settings (Scale & Settings) -> Time to Live -> **On (no default)** -> Save. No bloquea (la app verifica expiracion en codigo), solo habilita auto-limpieza.
- **Verificacion sin/ con header:** `/health` es liveness (200 siempre). `/health/ready` exige `X-Diag-Key`; en navegador (sin header) responde 404 a proposito. `503` con desglose `faltante/error` por componente; `wa-token`/`wa-verify-token`/`PhoneNumberId` en `faltante` son esperados sin WhatsApp real.
- **PowerShell:** `curl` es alias de `Invoke-WebRequest` (no acepta `-H`); usar `curl.exe` o `Invoke-RestMethod -Headers`. `Invoke-RestMethod` lanza excepcion en 503 y oculta el cuerpo; `curl.exe -s` lo imprime.
